# Part I — 오프라인 골든 스냅샷 검증 (§1~7)

## 1. 문제의식 / 가설

**기존 동작**: AI 워크플로우 생성 시, 사용자의 누적 개인 패턴(memory)을 `load_memory`로 **전부 프롬프트에 덤프**했다. 패턴이 쌓일수록 발화와 무관한 정보까지 LLM 컨텍스트를 오염시킨다.

**가설**: 발화와의 의미 유사도(BGE-M3 임베딩)로 **관련 패턴만 골라 주입(RAG)** 하면, 정답률은 유지하면서 노이즈를 줄일 수 있다. 무관한 잡담은 아예 주입하지 않아야(게이트) 한다.

**검증 질문 3개**
1. RAG가 정답 패턴을 놓치지 않는가? (hit율)
2. RAG가 노이즈를 실제로 줄이는가? (ON vs OFF 노이즈)
3. 무관 발화를 차단하는가? (distractor 게이트)

---

## 2. 실험 설계 (방법론)

| 구성 | 내용 |
|------|------|
| 페르소나 corpus | 가상 사무직 직장인의 누적 업무 패턴 **6종** (HWP 문서작성 / Outlook 메일 / 주간보고 / 구글시트 / Notion 회의록 / Slack 공지) |
| 발화 시나리오 | **7개** — 업무 발화 6 + distractor 1 |
| 정답지(gold) | 발화마다 떠야 마땅한 패턴을 사전 매핑 (primary/secondary) |
| 임베딩 | 실제 **BGE-M3, 768차원** (1024 → Matryoshka 절단, 프로젝트 전역 SSOT, pgvector `vector(768)`와 동일) |
| 비교군 | **ON** = RecallPersonalSkillsUseCase(코사인 top-k + min_score 게이트) vs **OFF** = `load_memory` 전체 덤프(기존) |

### 핵심 개념 — distractor (음성 대조군)
RAG가 "잘 작동한다"를 증명하려면, 정답이 떠야 하는 발화뿐 아니라 **"아무것도 뜨면 안 되는 발화"** 도 테스트해야 한다. 본 실험의 distractor는 **"점심 뭐 먹는 게 좋을까"** — 업무와 전혀 무관한 잡담이다. 제대로 된 RAG라면 관련 점수가 모두 낮아 `min_score` 게이트에서 걸러져 **빈손(∅)** 을 반환해야 정답이다. (기존 전체덤프는 잡담에도 업무 패턴 6개를 전부 주입한다.) distractor는 "RAG가 노이즈를 거르는 능력"을 증명하는 장치다.

### 엔지니어링 포인트 — 골든 임베딩 스냅샷 방식
실제 BGE-M3 벡터를 **1회만** 떠서 JSON fixture로 커밋 → 이후 평가/CI는 이 스냅샷을 결정론적으로 재생. **GPU·Modal 없이도** 의미검색 품질을 회귀 검증한다. (캡처 = `capture_embeddings.py`, 재생 = `SnapshotEmbedder`)

## 3. 측정 결과 (실측)

### 3-1. ON vs OFF 비교 (top_k=3, min_score=0.5)

| 지표 | RAG ON | OFF(전체덤프) |
|------|:------:|:------:|
| hit율 (정답 top-k 진입) | **100%** | 100% |
| 평균 노이즈 (무관 패턴 주입 수) | **0.83** | **4.83** |
| distractor("점심") 반환 | **∅ (게이트 차단)** | 6개 전부 덤프 |

→ **정답률 동등(100%)하게 유지하면서 노이즈 4.83 → 0.83, 약 5.8배 감소** (4.83 ÷ 0.83 = 5.82).

### 3-2. 발화별 상세 (ON, top_k=3 / min 0.5)

| 발화 | hit | ON 노이즈 | OFF 노이즈 |
|------|:---:|:---:|:---:|
| 이번 주 주간 보고서 자동으로 작성해줘 | ✅ | 0 | 4 |
| 보고서 문서 하나 만들어줘 | ✅ | 2 | 5 |
| 고객사에 안내 메일 발송 자동화 | ✅ | 0 | 5 |
| 이번 달 매출 데이터 정리해서 표로 | ✅ | 1 | 5 |
| 회의 끝나면 회의록 정리해서 저장 | ✅ | 2 | 5 |
| 팀 전체한테 공지 하나 돌려줘 | ✅ | 0 | 5 |
| 점심 뭐 먹는 게 좋을까 (distractor) | — | 0 (∅) | 6 |

### 3-3. min_score / top_k 스윕 (전체 실측)

| top_k=3 기준 | min 0.3 | min 0.4 | min 0.5 | min 0.6 | min 0.7 |
|:---:|:---:|:---:|:---:|:---:|:---:|
| hit율 | 100% | 100% | **100%** | 50% | 17% |
| 노이즈 | 1.83 | 1.83 | **0.83** | 0.17 | 0.00 |
| distractor 게이트 | ❌ FAIL | ✅ OK | ✅ OK | ✅ OK | ✅ OK |

(top_k=2: min0.5에서 hit 100% / 노이즈 0.50 · top_k=5: hit 100% / 노이즈 1.17 — hit율·게이트는 top_k 무관하게 min_score가 지배)

**판독**
- **min_score < 0.4 금지** — 0.3은 distractor 게이트가 뚫려(잡담에 패턴 주입) 전부 FAIL.
- **0.4~0.5 = 안전대** — hit 100% + distractor 차단 동시 충족.
- **0.6 이상 = 정답 손실** — hit율 50%(0.6) → 17%(0.7)로 급락(정답까지 게이트에 걸림).

### 3-4. 코사인 점수 분포 (분리대, 실측)

| 구분 | 코사인 |
|------|:---:|
| 정답(primary) 최저 | **0.553** (`doc_tool_hwp` ← "보고서 문서 하나 만들어줘") |
| 정답(primary) 최고 | **0.754** (`meeting_notes_notion` ← "회의록 정리") |
| distractor 최고 | **0.372** ("점심" ↔ `report_schedule_weekly`) |
| **분리 마진** | **정답 최저 0.553 − distractor 최고 0.372 = +0.181** |

→ 정답군과 잡담군이 **+0.181 마진으로 깨끗이 분리**. min_score 0.5는 그 사이 골을 통과한다.

## 4. 핵심 의사결정

### min_score = **0.5 확정** (2026-06-03)
- 0.4~0.5 안전대 중 **노이즈 최대 차단** 쪽. 코드 기본값(`recall_personal_skills_use_case.py:15`)과 일치 → **코드 변경 없음**.
- 트레이드오프 인지: 정답 최저(0.553)와 게이트(0.5) 헤드룸이 +0.053로 얇음 → corpus가 크게 늘면 일부 약한 정답이 게이트에 걸릴 수 있어, 재측정 트리거로 남겨둠.

### top_k = **3 권장 (안전마진)**
"보고서 문서 하나 만들어줘" 발화의 코사인 순위(실측):

| 순위 | 패턴 | 코사인 | |
|:---:|------|:---:|---|
| 1 | report_schedule_weekly | 0.608 | 비정답("보고서" 단어 반응) |
| 2 | **doc_tool_hwp** | **0.553** | **← 정답(primary)** |
| 3 | meeting_notes_notion | 0.548 | 비정답 |

- **top_k=1 불가**: 정답이 1위 비정답에 밀려 2위 → top_k=1이면 hit 깨짐.
- **top_k=2는 현 corpus 통과하나 마진 취약**: 정답(2위 0.553)과 3위(0.548) 간격이 **단 0.005** — 패턴이 조금만 늘어 정답이 3위로 밀리면 잘림.
- → 안전상 **top_k=3**. (노이즈 0.83은 OFF 4.83 대비 여전히 5.8배 낮음.) 단정적 "필수"가 아니라 마진 근거의 "권장"이 정확.

# Part II — 실적용(라이브) 테스트 (§8)

## 8.0 목적과 방식

오프라인(§1~7)은 `InMemoryPersonalMemoryStore` + `SnapshotEmbedder`로 **검색 품질만** 결정론적으로 측정했다(GCS I/O·Modal 호출 제외). Part II는 **운영과 동일한 경로**를 실제로 태워 그 결과가 라이브에서도 재현되는지 확인한다.

누적방식은 **두 갈래**로 합의됐다(2026-06-05 카톡):
- **(2번) 직접 시드 — 본 절에서 측정**: 운영 구현체 `GCSMemoryStore.save_file/save_embedding`를 그대로 호출해 실제 GCS에 `.md` + `.emb.json`을 적재하고, 임베딩은 스냅샷이 아닌 **실 Modal BGE-M3**로 생성. 이후 라이브 `recall`(GCS load + Modal embed)로 검색. 저장-타이밍 트리거(배선)에 의존하지 않아 검색 충실도를 독립적으로 닫는다.
- **(1번) 실파이프라인 경유 — 대원님 배선(Phase D) 완료 후 별도 측정**: 저장 트리거까지 실제로 거치는 end-to-end. 본 리포트엔 배선 완료 후 추가 기록 예정.

재현 러너: `live_eval.py` (`seed`/`recall`/`cleanup`). 대상은 전용 eval user_id(고정 uuid5 `1f69e012-…4c94`)로, 실유저 데이터와 격리된다.

## 8.3 핵심 발견 — 라이브 ≡ 오프라인 (소수점까지 일치)

실 GCS 직렬화·파싱 왕복(`_serialize_md_file ↔ _parse_md_file`)과 실 Modal BGE-M3 임베딩을 거쳐도 **코사인 점수가 골든 스냅샷과 소수점 3자리까지 동일**(0.553·0.754·0.372·마진 +0.181)했다. 이는 두 가지를 함께 입증한다.
1. **골든 스냅샷의 대표성** — 오프라인 결정론 검증이 라이브 동작을 그대로 반영한다(= §1~7 결론을 라이브가 추인).
2. **GCS 저장 경로 무손실** — 직렬화/역직렬화가 본문·임베딩을 변형 없이 보존한다.

latency도 0.32s/발화로, recall이 의도분석 직후 동기 주입돼도 체감 지연이 없다.

## 8.4 충실도 노트 — 실누적은 "워크플로우 단위"

staging 버킷에 이미 존재하는 실유저(`bec5b780-…`, uuid5)의 누적 파일은 **`workflow_{uuid}.md`** 형태였다. 즉 실파이프라인(1번/Phase D)은 본 검증의 의미명 corpus(`doc_tool_hwp.md` 등)와 달리 **워크플로우 1건 = 파일 1개** 단위로 LLM 요약을 적재한다. 검색 로직(`load_index`→코사인→top_k)은 파일 명명과 무관하게 동일하게 동작하므로 본 절의 결론에 영향은 없으나, 1번 측정 시 corpus 입도(per-pattern)와 실누적 입도(per-workflow)의 차이를 변수로 인지해야 한다.

## 8.5 결론 / 잔여

> 라이브(실 GCS + 실 Modal BGE-M3) end-to-end 검색이 오프라인 골든과 **소수점까지 일치**(hit 100% / 노이즈 0.83 / 마진 +0.181 / distractor ∅), latency 0.32s/발화. 운영 설정 **min_score 0.5 / top_k 3**의 라이브 타당성이 확인됐다.

**잔여(1번, 대원님 배선 후)**: 저장 트리거까지 거치는 실파이프라인 end-to-end 1회 → 본 절에 추가. 그 전까지 프론트 실적용 레그(RAG 주입 ON/OFF 비교)는 `supervisor.py`/`composer_graph.py` 배선이 선행돼야 측정 가능(§7 배선 현황 참조).
